# AstrBot 插件：模拟修仙 ver.M

> **版本:** ver.M 1.1  
> **许可证:** AGPL-3.0  
> **魔改作者:** MAMERAKKKKON  
> **架构:** Clean Architecture  

`ver.M` 是 MAMERAKKKKON 在现有 AstrBot 修仙插件基础上持续修改、扩展和平衡的魔改版本，版本号从 `1.1` 起独立记录。

本插件提供群聊修仙模拟玩法。玩家可以修炼、战斗、探索、炼丹、轮回并争夺传承，从凡人一步步踏上仙途。

---

## 🧭 ver.M 魔改更新进程

以下内容以接手魔改前的原版文件为基线，通过逐文件代码与配置对比整理。截至 `ver.M 1.1`，在排除运行缓存后，共修改了 40 个原版文件，并新增了 1 个炼丹经济维护脚本；没有删除原版功能文件。

### 第一阶段：独立版本与群聊管理

- 项目正式更名为 **模拟修仙 ver.M**，由 **MAMERAKKKKON** 维护，版本号从 `1.1` 起独立记录。
- 保留内部插件标识 `astrbot_plugin_monixiuxianv3`，避免升级后无法读取原有服务器数据。
- 启用全插件群聊白名单：白名单为空时保持原版行为；填写群号后，插件命令仅能在指定群聊使用，同时阻止私聊绕过限制。
- 统一帮助文本、插件元数据和运行日志中的名称、作者及版本标识。

### 第二阶段：轮回与传承体系重做

- 原「弃道重修」更名为 `轮回转世`，增加二次确认、空闲状态检查、贷款检查和 60 秒冷却。
- 新增**永久传承池**与**本世传承池**：传承挑战所得先进入本世池，轮回时再并入永久池，新角色创建后自动继承。
- 炼气期轮回不产生境界奖励；从筑基开始，按筑基、金丹、元婴、化神、炼虚、合体、大乘、渡劫八档结算奖励。
- 境界奖励同时包含 HP、攻击、防御、暴击、爆伤等百分比，以及 HP、攻击、防御、神识白值。
- 百分比传承改为**每一世乘算叠加**，固定白值继续加算，既强化多世成长，也避免白值指数膨胀。
- 传承挑战接入统一战斗引擎，可争夺 HP、攻击、MP、防御、暴击率、爆伤及四类白值。
- 传承积累越高，后续夺取效率越低：50%、100%、200%、500%分别进入不同衰减区间。
- 挑战胜利后收益进入本世池，被挑战者同时损失部分传承；挑战失败损失当前修为的 1%。
- 新增传承信息和排行榜展示，可分别查看永久池、本世池与转世次数。

### 第三阶段：死亡结算与特殊道具

- 突破失败但未死亡时，统一损失当前修为的 10%；突破死亡不再删除整个角色，而是保留角色并清空修为和灵石。
- 秘境死亡同样保留角色，清空修为和灵石，并在结算文本中显示实际损失。
- 新增神品丹药 **大番茄**：服用后获得 1 次持续免死，效果不会按时间消失，只有真正掷中死亡判定时才会消耗。
- 大番茄可在突破死亡与秘境死亡中触发；触发后原事件继续按照“未死亡”分支结算，不会把普通突破失败惩罚写死在道具逻辑中。
- 新增全服唯一的神秘专属法器，禁止商店刷新并限制实际使用者；百科中的 ID、品级、类型、价格和境界要求均以遮蔽文本显示，隐藏效果不在公开文档披露。

### 第四阶段：统一战斗系统

- 重构战斗服务，统一生成真实战斗快照，纳入修炼类型、当前属性、装备、物防、法防以及传承暴击/爆伤。
- 灵修按照法术伤害对方法防御结算，体修按照物理伤害对物理防御结算。
- 防御公式调整为攻防相对减伤，避免后期旧公式无限接近 100% 减伤，并保证每次攻击至少造成 1 点伤害。
- 暴击率为 0 时不再误触发暴击；基础暴击伤害为 1.5 倍，并可受到永久传承加成。
- 切磋与决斗共用稳定的目标解析、状态检查、战斗日志和 60 秒冷却，修复部分消息适配器下偶发无法识别 `@目标` 的问题。
- 当前阶段的决斗与切磋规则完全相同：均为无损战斗，最多 20 回合，超时后按剩余生命比例判定胜负。
- 传承挑战和世界 Boss 已接入同一套回合战斗核心，减少各玩法各算一套属性造成的结果偏差。
- Boss 血量改为跨挑战真实保留；玩家以当前血量参战，获胜保存实际剩余血量，战败固定保留 1 点血量，因此回血丹药具备实际用途。

### 第五阶段：宗门与洞天福地完善

- 宗门系统补全统一的成员、职位和权限检查，宗主身份优先以宗门数据反查，减少职位数据不一致。
- 创建宗门要求境界索引 3、消耗 10,000 灵石；宗门上限 50 人，初始建设度和资材均为 100。
- 宗门捐献调整为 `1 灵石 = 10 建设度 + 1 个人贡献`。
- 宗门任务增加 1 小时冷却，每次获得 10～30 贡献，并为宗门增加对应贡献 10 倍的资材。
- 新增宗门成员列表、贡献展示、职位管理和 `解散宗门`；宗主退出时会正确处理仅剩一人的宗门。
- 洞天改为「小洞天 → 中洞天 → 大洞天 → 福地 → 洞天福地」逐阶进化，并保存等级、修炼加成、灵石产出和收取时间。
- 洞天福地达到最高形态后仍可额外强化 50 次：首次花费 100 万灵石，此后每次递增 50 万；每级增加 5% 修炼加成和每小时 1,000 灵石产出。
- 洞天修为收益增加每小时 50,000 的上限，防止高修为与长时间离线组合造成数值爆炸。

### 第六阶段：商店、百科与储物戒体验

- 新增 `商店` 指令，可在一条消息中一次性查看丹阁、器阁和百宝阁，原三个独立指令继续保留。
- `查看物品` 改为全局物品百科，不再要求玩家先持有道具；支持完整名称查询、关键词候选、物品 ID、品质、类型、价格、境界、效果和介绍展示。
- 物品效果展示统一支持气血、修为、属性、寿命、突破率、免死次数、装备伤害、防御和修炼倍率等字段。
- 修复储物戒对配置型丹药的分类，并让突破丹、普通丹药和特殊丹药统一进入丹药分类。
- 重写赠予目标和数量解析，兼容不同适配器的 At 字段、纯 QQ 号及 `×/x/*` 数量写法。
- 赠予改为直接送达对方储物戒；若接收失败会自动返还发送者物品，避免中途异常导致道具丢失。

### 第七阶段：天地灵眼自动化

- 插件启动后立即检查灵眼，此后每小时检查一次；无主灵眼不足时自动生成，最多同时保留 5 个无主灵眼。
- 完善 `灵眼信息` 展示，可查看自己的待收取收益以及最多 5 个可抢占灵眼。
- 新增管理员命令 `生成灵眼`，用于手动补充和排查自动生成系统。
- 插件关闭时会主动停止灵眼后台任务，避免热重载后产生重复定时器。

### 第八阶段：炼丹经济全面重整

- 修复原版配方中的炼制费只展示、不扣除问题；现在每次尝试都会真实扣除材料和灵石，失败同样消耗本次资源。
- 批量炼丹同时受到材料和灵石限制，资源不足时只炼制实际可承担的次数，并显示真实总费用。
- 对全部 59 张配方重新计算炼制费和成丹参考价，消除后期手续费高出材料价值数百倍的断层。
- 炼制费按品质约占材料价值的 25%～60%；回血丹额外增加 15%，破境丹额外增加 10%；目标上限为材料价值的 75%，整数取整后实际最高约为 75.5%。
- 在最低要求炼丹等级下，材料和费用按成功率折算后的期望成本约为成丹参考价的 65%～70%；炼丹等级越高，成功率和利润空间越高。
- 圣品成丹的商店权重降至最多 20；帝品、道品、仙品和神品成丹的商店权重全部降为 0，高阶成丹改以玩家炼制和坊市流通为主。
- 高阶炼丹材料仍保留低概率商店来源，使玩家可以购买材料参与炼丹，但无法靠蹲点直接购买高阶成丹。
- 新增炼丹经济重算脚本，便于以后调整材料价格后统一复算全部配方。

### 当前阶段状态

- `ver.M 1.1` 已完成轮回传承、核心战斗、Boss 持久血量、死亡与免死、商店百科、灵眼自动化、宗门洞天和炼丹经济的首轮重整。
- 功法与装备的 `exp_multiplier` 已接入真实闭关结算；丹药 `add_spiritual_power` 已按灵修灵气/体修气血写入当前值与上限，不再只显示文案。
- 精神力（MP）的丹药增减与战斗用途暂缓设计，避免在用途未确定前提前固化数值公式。
- 决斗目前有独立命令与日志类型，但玩法暂时等同无损切磋；后续可在统一战斗核心上继续设计真正的风险与奖励。
- 数值仍会根据服务器实际玩家产出、市场成交和战斗记录持续调整。

**炼丹职业系统**

- 独立的炼丹职业系统，不依赖修炼境界
- 炼丹等级：0-100级，每10级获得新称号
- 成功率加成：每级+0.5%（最高+50%）
- 经验获取：炼丹成功获得经验，失败获得1/3经验
- 11个炼丹称号：见习炼丹师 → 初级炼丹师 → 中级炼丹师 → 高级炼丹师 → 炼丹大师 → 炼丹宗师 → 炼丹圣手 → 丹道真人 → 丹圣 → 丹帝 → 丹神

**配方系统**

- 59个完整配方（33种常规丹药 + 26种突破丹药）
- 配方包含：所需材料、炼丹等级要求、基础成功率、炼制费用
- 每次尝试都会实际扣除材料和炼制费，失败仍会消耗本次资源
- 最低所需炼丹等级下，期望炼制成本约为成丹参考价的70%；等级越高，成功率和利润空间越高
- 圣品成丹仅在丹阁极低概率刷新，帝品及以上成丹不再刷新，主要由玩家炼制和坊市流通
- 材料品质与丹药品质相匹配

### 装备系统详解

**装备类型与属性**

- **武器（96种）**：提供物理伤害、法术伤害、防御、精神力等属性
- **法器/防具（10种）**：提供防御、气血上限、精神力等属性
- **功法（10种）**：提供全面属性加成和修炼倍率加成（exp_multiplier）

**装备属性显示**
- 玩家信息显示格式：`总属性(基础+装备)`
- 例如：`法伤：105(100+5)` 表示基础法伤100，装备加成5
- 战斗系统自动计算装备加成

**装备等级需求**
- 所有装备都有 `required_level_index` 字段，对应36境界系统
- 境界不足无法装备高级装备

### 丹药系统详解

**丹药分类**

1. **破境丹（27种）** - 专门用于突破境界
   - 位于 `pills.json` 配置文件
   - 提供突破成功率加成（breakthrough_rate）
   - 价格范围：10k灵石 - 200亿灵石
   - 品质：灵品 → 珍品 → 圣品 → 帝品 → 道品 → 仙品 → 神品

2. **增益丹（33种）** - 提升属性和修为
   - 位于 `items.json` 配置文件
   - 效果多样：恢复气血、增加修为、提升属性、突破加成等
   - 部分丹药有副作用（如焚天逆命增益丹扣除寿命）

**丹药效果显示**
- 效果：显示数值加成（如"恢复气血+100"、"突破成功率+40%"）
- 介绍：显示描述文本（如"辅助修士突破至筑基期的灵丹"）
- 储物戒和商店都会分别显示效果和介绍

**特殊丹药**
- **焚天逆命增益丹**：提升18%突破率，但会损失800气血和10年寿命
- **寿命系统**：部分丹药可增加/减少寿命，寿命变化会在服用时显示

### 储物戒系统详解

**储物戒等级**

储物戒按容量和品质分为8个等级，需要达到对应境界才能升级：

| 储物戒名称 | 品质 | 容量  | 需求境界 | 价格      |
| ---------- | ---- | ----- | -------- | --------- |
| 基础储物戒 | 凡品 | 20格  | 无       | 0（初始） |
| 灵玉储物戒 | 凡品 | 30格  | 筑基初期 | 1,000     |
| 玄铁储物戒 | 珍品 | 50格  | 金丹初期 | 5,000     |
| 星辰储物戒 | 珍品 | 80格  | 元婴初期 | 15,000    |
| 乾坤储物戒 | 圣品 | 120格 | 炼虚初期 | 50,000    |
| 须弥储物戒 | 圣品 | 200格 | 合体初期 | 200,000   |
| 混元储物戒 | 帝品 | 500格 | 大乘初期 | 1,000,000 |
| 太虚储物戒 | 神品 | 999格 | 地仙初期 | 9,999,999 |

**升级机制**
- 使用 `升级储物戒` 命令自动升级到下一级
- 系统自动检测境界和灵石是否满足条件
- 只能升级，不能降级或卸下
- 升级后容量立即生效，已有物品保留

**物品分类**
- 丹药：所有包含"丹"字的物品（不含内丹）
- 材料：草、铁、石、沙、参、芝、果、花、根、子、髓、核、粉、砂、皮、血、息、齿轮、碎片、精华、残页、遗物、种子、内丹
- 法器：剑、刀、枪、弓、幡、甲、袍、铠、阵
- 功法：功、诀、经、法
- 储物戒：储物戒类物品
- 其他：不属于以上分类的物品

**容量计算**
- 每种物品占1格，无论数量多少
- 例如：100个灵草只占1格，1个灵草也占1格
- 容量满时无法获得新种类物品，但可以增加已有物品数量

### 功法残篇合成系统

功法可以通过收集残篇自动合成完整版本，不同品质的功法需要的残篇数量不同：

| 功法品质  | 合成规则 | 功法示例                             |
| --------- | -------- | ------------------------------------ |
| 凡品      | 5个残篇  | 长春功、御风诀                       |
| 珍品      | 10个残篇 | 不动明王经、北冥神功、九阳神功       |
| 圣品/帝品 | 15个残篇 | 焚天诀、道经、吞天魔功、他化自在大法 |

**合成机制：**
- 当你获得功法残篇时，系统会自动检测是否集齐所需数量
- 集齐后自动消耗残篇并合成完整功法，无需手动操作
- 完整功法也可以直接掉落，但概率极低（鼓励通过收集残篇合成）
- 功法残篇主要通过秘境探索和历练系统的稀有掉落获得

## 🏗️ 技术架构

本插件采用 **Clean Architecture（整洁架构）** 设计：

| 层级                             | 职责                     | 示例                         |
| -------------------------------- | ------------------------ | ---------------------------- |
| **表现层（Presentation）**       | 处理用户命令和消息格式化 | PlayerHandler、CombatHandler |
| **应用层（Application）**        | 实现业务逻辑和用例       | PlayerService、CombatService |
| **领域层（Domain）**             | 定义核心业务模型和规则   | Player、Sect、Boss模型       |
| **基础设施层（Infrastructure）** | 提供数据持久化和外部服务 | PlayerRepository、Database   |

**架构优势：**

- 清晰的分层结构，易于维护和扩展
- 依赖注入容器管理对象生命周期
- 领域模型与数据库解耦
- 业务逻辑与表现逻辑分离
